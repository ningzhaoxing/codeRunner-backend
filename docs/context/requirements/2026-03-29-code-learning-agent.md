# 代码学习 Agent — 产品需求文档v1.0（PRD）

> **状态**：草稿 · **日期**：2026-03-29 · **作者**：内部

---

## 1. 背景与目标

### 背景

CodeRunner 已为博客平台提供代码执行能力，读者可以在文章中直接运行代码片段并看到输出结果。然而，"能运行"和"能理解"之间还有很大的鸿沟：

- 读者遇到报错时，只能看到错误信息，不知道原因
- 读者看到看不懂的代码，只能离开页面去搜索
- 读者想验证某个函数的边界情况，需要自己手写测试代码
- 博客中的代码片段通常是不完整的（缺少 import、main 函数等），读者想运行时无从下手

**代码学习 Agent** 的目标是补全这一缺口——让读者在不离开文章页面的前提下，通过自然语言对话调试、理解、验证文章中的任意代码。

### 目标

- 降低技术博客读者的代码理解门槛
- 让"看代码"和"跑代码"之间形成自然闭环
- MVP 阶段专注单篇文章内的对话体验，不引入跨文章复杂性

### 不是什么

代码学习 Agent **不是**代码编辑器，不是代码补全工具，也不是通用 AI 助手。它的感知范围严格限定在当前打开的这篇文章。

## 2. 功能能力清单

| # | 能力 | 描述 | 触发方式 |
|---|------|------|---------|
| F1 | **代码解释** | 解释文章中某段代码的逻辑、设计意图和行为 | 用户自然语言提问 |
| F2 | **调试辅助** | 分析用户描述的报错或非预期输出，给出修复建议 | 用户自然语言提问 |
| F3 | **测试生成** | 为指定代码块生成边界测试用例代码 | 用户自然语言提问 |
| F4 | **执行确认** | Agent 生成可运行代码后，在对话中提议执行并等待确认；用户通过**自然语言回复**（如"好的"、"运行吧"、"确认"）触发执行，无需额外按钮 | Agent 主动提议 + 用户对话回复确认 |
| F5 | **流式回复** | AI 回答通过流式方式逐字推送，用户不需要等全部生成完才看到内容 | 自动 |
| F6 | **多轮对话** | 在同一篇文章的会话内，支持连续追问，Agent 记住上下文 | 自动（session TTL 1 小时） |
| F7 | **代码片段补全至可运行** | 博客代码块通常是不完整的片段；Agent 在提议执行前，自动推断并补全缺失部分（包声明、import、main 函数包装、依赖类型定义等），生成完整可运行代码，并向用户说明补了什么 | Agent 判断代码块不可直接运行时自动触发 |

### 能力边界说明

- F1/F2/F3 均基于文章内已有的代码块，Agent **不能**访问文章外的代码
- F4 的执行必须经过用户确认，Agent **不能**自动执行任何代码；用户通过对话回复确认，Agent 识别确认意图后触发执行；若当前有多个待确认 Proposal，Agent 在执行前应向用户确认是哪一个
- F6 的上下文仅限当前文章（article 级别），关闭页面或 TTL 到期后会话结束
- F7 补全的代码必须与原代码块语义一致，Agent **不能**修改原有逻辑，只能补充运行所需的脚手架代码；补全说明（description）需清晰列出新增内容

## 3. 典型场景举例

### 场景一：调试报错

> 读者在博客文章中运行了一段 Go 代码，得到 `index out of range` panic。

**用户行为**：在对话框输入"这段代码为什么会 panic？"

**Agent 行为**：

1. 结合文章中嵌入的代码内容，分析数组访问越界的原因
2. 给出修复后的完整代码，并说明改动内容（如"将循环条件改为 `i < len(s)-1`"）
3. 调用 `propose-execution`，在对话中展示修复后的代码并说明改动，提示"如需运行请回复确认"

**用户行为**：回复"好的，运行吧"

**系统行为**：Agent 识别确认意图，通过 CodeRunner 执行修复后的代码，将 stdout/stderr 推送回对话

**用户行为**：追问"为什么 `len(s)-1` 能解决这个问题？"

**Agent 行为**：结合刚才的执行结果和文章上下文继续解释

---

### 场景二：理解代码

> 读者看到一段使用 `sync.WaitGroup` 的 goroutine 代码，不明白为什么要这样写。

**用户行为**：在对话框输入"为什么这里要用 WaitGroup，不用行不行？"

**Agent 行为**：

1. 直接引用文章中对应的代码块
2. 解释 WaitGroup 的作用——等待所有 goroutine 完成后再继续
3. 说明不用时会发生什么（main 函数提前退出，goroutine 被强制终止）
4. 无需调用任何工具，直接给出文字解释

---

### 场景三：生成并验证测试

> 读者实现了一个 `fibonacci(n int) int` 函数，想验证边界情况。

**用户行为**：在对话框输入"帮我生成几个边界测试用例"

**Agent 行为**：

1. 引用文章中的 fibonacci 函数代码
2. 生成包含 `n=0`、`n=1`、`n<0`、`n=40`（大数）等边界 case 的测试代码
3. 提议执行测试代码，在对话中展示测试代码并提示"如需运行请回复确认"

**用户行为**：回复"跑一下"

**系统行为**：Agent 识别确认意图，运行测试代码，将测试输出推送回对话

**Agent 行为**（可选追问）：用户问"这个负数 case 你是怎么处理的？" → Agent 解释测试中的错误处理逻辑

---

### 场景四：运行不完整的代码片段

> 文章中展示了一段 Go 代码片段，只包含一个函数定义，没有 `package main` 和 `main()` 入口，也没有 import。读者好奇这段代码运行起来会输出什么。

**用户行为**：在对话框输入"帮我跑一下这段代码看看输出"

**Agent 行为**：

1. 识别该代码块为不完整片段（缺少 `package main`、`import`、`main()` 包装）
2. 自动补全成可运行的完整程序：添加 `package main`、必要的 import、以及调用该函数并打印输出的 `main()` 函数
3. 通过 `propose-execution` 在对话中展示**完整代码**，列出新增内容（如"补充了 `package main`、`import fmt`，以及在 `main()` 中调用并打印结果"），提示"如需运行请回复确认"

**用户行为**：回复"可以，运行"

**Agent 行为**（可选追问）：用户问"你是怎么知道要调用这个函数的？" → Agent 解释补全逻辑

**补全范围**（Agent 可推断补充的内容）：

- 包声明（`package main`）
- import 语句（根据代码中用到的标准库推断）
- `main()` 函数包装（调用片段中的函数并打印结果）
- 片段中引用到的简单 struct / type 定义（若原文章中其他代码块有定义则优先复用）

**补全边界**（Agent 不做的事）：

- 不修改原有函数逻辑
- 不引入文章未提及的第三方库
- 若无法合理推断如何调用（如函数需要复杂外部依赖），Agent 说明原因并询问用户提供更多信息

## 4. 用户交互流程

### 4.1 完整流程（文字描述）

```
[用户打开文章]
    ↓
[前端初始化：将文章内容和所有代码块发送给 Agent，建立 SSE 连接]
    ↓ (后台静默，用户无感知)
[用户在对话框输入问题并发送]
    ↓
[Agent 流式输出回答，用户看到文字逐字出现]
    ↓
    ├── [纯文字回答] → 显示完整回复，用户可继续提问
    │
    └── [Agent 提议执行代码]
            ↓
        [对话区出现代码块 + 修改说明文字 + 提示"如需运行请回复确认"]
            ↓
            ├── [用户回复确认（"好的"/"运行"/"跑一下"等）]
            │       → Agent 识别确认意图 → 后台异步执行 → 结果推送到对话区
            │       ↓
            │   [用户继续追问或结束]
            │
            └── [用户转移话题 / 不回复] → 对话继续，Proposal 10 分钟后过期
```

### 4.2 SSE 连接生命周期

- SSE 连接在 Agent 回复完成（`done` 事件）后**保持打开**，等待异步执行结果
- 用户发起新一条消息时，旧连接收到 `interrupted` 事件后被新连接替代
- 用户离开文章页面时，前端主动关闭连接
- Session TTL 1 小时，到期后会话数据清除

### 4.3 前端展示要点（指引性，非强制 UI 规范）

| 状态 | 建议展示 |
|------|---------|
| Agent 回复中 | 流式文字 + 光标动画 |
| Agent 提议执行 | 代码块（带语言标注）+ 修改说明文字 + "如需运行请回复确认"提示 |
| 等待执行结果 | 对话区显示"执行中…"的 Agent 消息或 loading 占位 |
| 执行成功 | 展示 stdout/stderr 输出，保留在对话历史中 |
| 执行失败 | 展示错误信息，允许用户继续追问 |
| 被打断（interrupted） | 对话区提示"上一条回复已被打断" |

## 5. 成功指标与验收标准

### 性能指标

| 指标 | 目标值 | 说明 |
|------|--------|------|
| 首字时间（TTFT） | ≤ 2s | 从用户发送消息到第一个字符出现 |
| 完整回复时间 | ≤ 15s | 普通问题；复杂推理可适当放宽 |
| 确认意图识别延迟 | ≤ TTFT（同普通消息） | 用户发送确认消息后，Agent 识别并开始触发执行的时间 |
| 执行结果推送延迟 | ≤ CodeRunner 执行时间 + 500ms | 执行完成后结果应立即出现 |

### 功能验收标准

| 功能 | 验收标准 |
|------|---------|
| 代码解释 | Agent 能准确引用文章中的代码块，不编造不存在的代码 |
| 调试辅助 | Agent 给出的修复代码在语法上正确，与原语言一致 |
| 测试生成 | 生成的测试代码覆盖至少 3 个边界 case，且语法正确 |
| 执行确认 | Proposal 有效期内确认可成功触发执行；过期后返回明确提示 |
| 多轮对话 | 同一 session 内，Agent 能引用 3 轮以上历史对话内容 |
| 上下文隔离 | 不同文章的 session 互不干扰 |
| 代码片段补全 | 补全后的代码语法正确可运行；说明中明确列出新增内容；原有函数逻辑不被修改 |

### 可观测性指标（内部）

- `agent_chat_duration_seconds`：单次 /chat 请求耗时分布
- `agent_tool_calls_total`：各工具调用次数及成功率
- `agent_sessions_active`：当前活跃 session 数

## 6. 异常场景用户侧表现

| 异常情况 | 用户看到什么 | 用户可以做什么 |
|---------|-------------|--------------|
| AI 接口调用失败（网络/API 限流） | 对话区显示"AI 服务暂时不可用，请稍后重试" | 重新发送消息 |
| AI 超过最大推理步数（10 步） | 对话区显示"分析超时，请换一种方式提问" | 重新提问或拆分问题 |
| gRPC 执行失败（CodeRunner 不可用） | 对话区显示"代码执行失败：{错误信息}" | 稍后再次确认，或放弃执行 |
| Proposal 已过期（10 分钟） | 点击"运行此代码"后提示"该提议已过期，请让 Agent 重新生成" | 重新提问让 Agent 再次生成代码 |
| Proposal 重复确认 | 第二次点击时提示"已提交执行，请等待结果" | 等待结果 |
| Session 过期（1 小时） | 发送消息后提示"会话已过期，请刷新页面重新开始" | 刷新页面 |
| SSE 连接断开（网络波动） | 前端重连（由前端实现）；若断连期间有执行结果，下次 /chat 时注入到对话上下文 | 继续提问，结果不会丢失 |
| 请求了不支持的编程语言执行 | Agent 回复"当前仅支持执行 Go/Python/JavaScript/Java/C 代码" | 切换语言或放弃执行 |
| 代码块 ID 不存在（前端 bug） | Agent 回复"未找到对应代码块，请确认你引用的代码在文章中" | 重新描述问题 |
| 代码片段无法合理补全（依赖过于复杂） | Agent 回复"这段代码依赖外部状态，我无法自动补全，请告诉我如何初始化 X" | 补充说明后重新请求 |

## 7. 不在范围内

| 功能 | 不做的原因 |
|------|-----------|
| 跨文章对话记忆 | 每篇文章上下文独立，混入其他文章会干扰推理质量；可在后续版本通过持久化 session 实现 |
| Agent 自动执行代码（无确认） | 代码执行有副作用，必须保留人工门控 |
| IDE 式代码自动补全（逐字提示） | 与对话式交互定位不同，是独立功能方向；F7 的"片段补全"仅限将不完整代码补全至可运行，不是 IDE 补全 |
| Agent 主动推送（执行完自动解释） | MVP 阶段保持用户驱动，Agent 不主动打断 |
| 博客前端具体 UI 实现 | 前端由博客系统团队负责，本文档仅提供指引 |
| Agent 微服务水平扩展 | 当前内存 session 不支持多实例共享，需引入 Redis 后支持，属于后续迭代 |
| 安全审查规则引擎 | 依赖 CodeRunner 现有的 Docker 沙箱隔离机制，不在 Agent 层重复实现 |
