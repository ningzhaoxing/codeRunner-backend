# 架构优化迭代方案

> 本文档记录当前架构的可优化点，作为后续迭代的参考路线图。  
> 更新时间：2026-04-01

---

## 优先级总览

| 优先级 | 方案 | 实现成本 | 预期收益 |
|--------|------|----------|----------|
| 🔴 高 | 容器池 | 中 | 执行延迟降低 50%+ |
| 🔴 高 | 结果缓存 | 低 | 重复请求毫秒级响应 |
| 🟡 中 | 异步执行队列 | 高 | 吞吐量大幅提升 |
| 🟡 中 | 代码预检查 | 中 | 安全性与资源利用率提升 |
| 🟡 中 | 接口限流防护 | 低 | 防止恶意攻击，保障服务稳定 |
| 🟢 低 | WebSocket 连接池 | 中 | 单 Worker 并发能力提升 |
| 🟢 低 | 热更新配置 | 低 | 运维便利性提升 |
| 🟢 低 | 分布式追踪完善 | 低 | 调用链可视化 |

## 1. 容器池 🔴

### 现状问题

每次代码执行都需要调用 `ensureContainerExists()` 检查容器状态，容器冷启动耗时 1-2s。当前架构中 5 种语言对应 5 个常驻容器，资源占用固定，无法弹性伸缩。

```
请求到达 → ensureContainerExists() → 容器执行 → 返回结果
                 ↑
            每次都要检查
```

### 优化方案

预先创建 N 个容器实例组成容器池，请求到达时从池中获取空闲容器，执行完成后归还而非销毁。

**核心数据结构：**

```
ContainerPool
├── idle    chan *Container   // 空闲容器队列
├── size    int               // 池大小
└── lang    string            // 语言类型
```

**生命周期：**

```
服务启动 → 预热 N 个容器放入 idle 队列
请求到达 → 从 idle 取出容器（blocking）
执行完成 → 重置容器状态 → 归还到 idle 队列
容器异常 → 销毁并补充新容器，保持池大小
```

**参数建议：**

| 语言 | 最小池大小 | 最大池大小 |
|------|-----------|-----------|
| Python | 2 | 8 |
| Go | 1 | 4 |
| JavaScript | 2 | 8 |
| Java | 1 | 3 |
| C | 1 | 4 |

### 预期收益

- 执行延迟降低 50%+（消除冷启动时间）
- 容器复用率提升，减少 Docker API 调用频率

## 2. 结果缓存 🔴

### 现状问题

相同代码重复执行浪费计算资源，在教学场景下尤为突出——大量学生提交相同作业时，每次都会触发完整的容器执行流程。

### 优化方案

对 `(language, codeBlock)` 组合计算 SHA256 指纹，命中缓存时直接返回结果，跳过容器执行。

**缓存键设计：**

```
cache_key = SHA256(language + ":" + codeBlock)
```

**执行流程：**

```
收到请求
    ↓
计算 cache_key
    ↓
查询 Redis → 命中 → 直接回调 callBackUrl（附带缓存标记）
    ↓ 未命中
容器执行
    ↓
写入 Redis（TTL: 1h）
    ↓
回调 callBackUrl
```

**Redis 数据结构：**

```
Key:   code:cache:{sha256}
Value: {"result": "...", "duration": 123, "cached": true}
TTL:   3600s
```

**不适合缓存的场景：**
- 包含随机数、当前时间的代码
- 读写文件的代码（副作用）
- 用户明确标记 `no_cache=true` 的请求

### 预期收益

- 重复请求响应时间从秒级降至毫秒级
- 高峰期容器资源压力显著降低

## 3. 异步执行队列 🟡

### 现状问题

当前调用链为同步模式：gRPC `Execute()` 需要等待 WebSocket 发送完成才返回。Worker 全忙时，新请求在负载均衡 `Get()` 阶段直接失败，没有排队缓冲，无法平滑处理请求峰值。

```
gRPC → P2C 选节点 → WebSocket Send → 返回 accepted
                        ↑
               Worker 全忙则直接失败
```

### 优化方案

引入消息队列，Server 收到请求后入队立即返回，Worker 从队列拉取任务异步执行。

**推荐技术选型：**

| 方案 | 优点 | 缺点 |
|------|------|------|
| Redis Stream | 已有 Redis 依赖可复用，部署简单 | 功能相对基础 |
| RabbitMQ | 支持优先级队列、死信队列 | 额外组件依赖 |
| Kafka | 高吞吐、持久化 | 运维成本高，小规模过重 |

**推荐 Redis Stream**（与结果缓存共用 Redis 实例）。

**新调用链：**

```
gRPC Execute
    ↓
入队 Redis Stream（key: code:queue）
    ↓
立即返回 {id, grpcCode: "accepted"}

Worker 消费者循环
    ↓
XREADGROUP 拉取任务
    ↓
容器执行
    ↓
HTTP 回调 callBackUrl
    ↓
XACK 确认消费
```

**支持优先级（可选）：**

```
普通用户 → code:queue:normal
VIP 用户  → code:queue:priority   （Worker 优先消费）
```

### 预期收益

- 系统具备削峰填谷能力
- Worker 全忙时请求排队而非失败
- 天然支持水平扩展（多 Server 共享同一队列）

## 4. 代码预检查 🟡

### 现状问题

恶意或低质量代码（死循环、fork 炸弹、危险系统调用）直接进入容器执行，只能依靠容器超时机制兜底，浪费资源且响应慢。

### 优化方案

在容器执行前增加静态检查层，快速拒绝明显危险或无效的代码。

**检查流程：**

```
收到请求
    ↓
语法检查（快速编译检测）
    ↓
黑名单检查（危险 API / 关键字）
    ↓
通过 → 进入执行队列
失败 → 立即返回错误，不消耗容器资源
```

**各语言检查策略：**

| 语言 | 语法检查 | 危险 API 检测 |
|------|---------|--------------|
| Python | `py_compile` | `os.system`, `subprocess`, `socket`, `eval(__import__)` |
| Go | `go/parser` AST 解析 | `os/exec`, `net`, `unsafe` |
| JavaScript | `acorn` AST | `require('child_process')`, `eval` |
| Java | `javac` 预编译 | `Runtime.exec`, `ProcessBuilder` |
| C | `gcc -fsyntax-only` | `system()`, `popen()`, `fork()` |

**复杂度限制：**
- 代码行数上限：500 行
- 嵌套深度上限：10 层
- 循环结构数量上限：20 个

### 预期收益

- 恶意代码在入口被拦截，减少容器资源浪费
- 语法错误快速返回，提升用户体验
- 结合缓存，避免缓存恶意结果

## 5. 接口限流防护 🟡

### 现状问题

代码执行接口对外暴露，没有任何限流保护。单个 IP 或 uid 可以无限制地发送请求，高频攻击会耗尽 Worker 资源，导致正常用户无法使用。

### 优化方案

在 gRPC 拦截器层增加限流，基于 `uid`（已在 proto 中携带）和客户端 IP 双维度限速。

**限流策略：**

```
按 uid 限流：每个用户每秒最多 N 次请求（令牌桶）
按 IP 限流：每个 IP 每秒最多 M 次请求（滑动窗口）
全局限流：整个服务每秒最多 K 次请求（漏桶兜底）
```

**实现方式（使用 Redis）：**

```
Redis Key: rate:uid:{uid}   TTL 自动过期
Redis Key: rate:ip:{ip}

超限时 gRPC 返回 codes.ResourceExhausted，附带 Retry-After 信息
```

**推荐参数（可通过热更新配置动态调整）：**

| 维度 | 默认限制 | 说明 |
| ------ | ---------- | ------ |
| 单用户 QPS | 5/s | 正常使用足够，防止单用户滥用 |
| 单 IP QPS | 20/s | 允许同一网络多用户并发 |
| 全局 QPS | 按 Worker 数 × 10 | 与 Worker 容量挂钩 |

**与代码预检查配合：**

```
请求到达
    ↓
限流检查（快，< 1ms）→ 超限立即拒绝，不进入后续流程
    ↓
代码预检查（静态分析）
    ↓
容器执行
```

### 预期收益

- 恶意攻击在入口被拦截，Worker 资源得到保护
- 正常用户不受影响
- 与热更新配置结合，限流阈值可动态调整无需重启

## 6. WebSocket 连接池 🟢

### 现状问题

Server 与每个 Worker 之间只维持一条 WebSocket 长连接，单连接故障或满载时无法快速切换，且单连接的并发能力受限于消息串行处理。

### 优化方案

Server ↔ Worker 建立多条 WebSocket 连接，构成连接池，请求在连接间轮询分发。

```
Server
├── Worker-A 连接池 [conn1, conn2, conn3, conn4]
├── Worker-B 连接池 [conn1, conn2, conn3, conn4]
└── Worker-C 连接池 [conn1, conn2, conn3, conn4]
```

**参数建议：**
- 每个 Worker 维持 2-4 条连接
- 连接健康检查间隔：10s
- 单连接故障后自动从池中摘除，异步重建

### 预期收益

- 单条连接故障不影响该 Worker 的其他请求
- 单 Worker 并发处理能力提升

## 7. 热更新配置 🟢

### 现状问题

修改 `config.yaml` 中的参数（如超时时间、容器资源限制、限流阈值）需要重启服务，在生产环境影响可用性。

### 优化方案

将可动态调整的配置存入 etcd（已有依赖），监听 Watch 事件实现热加载。

**适合热更新的配置项：**

```yaml
dynamic:
  execution_timeout: 30s        # 代码执行超时
  max_code_size: 65536          # 代码大小上限（字节）
  container_pool_size: 4        # 各语言容器池大小
  rate_limit_qps: 10            # 默认用户 QPS 限制
  callback_retry_count: 3       # 回调重试次数
  cache_ttl: 3600               # 缓存 TTL（秒）
```

**实现方式：**

```
服务启动 → 从 etcd 读取动态配置
          → 监听 etcd Watch 事件
          → 配置变更时原子替换（sync/atomic 或 RWMutex）
```

### 预期收益

- 参数调整无需重启，提升运维效率
- 支持灰度调整（先改一台观察效果）

## 8. 分布式追踪完善 🟢

### 现状问题

已在 gRPC 拦截器注入 TraceID（#14 已完成），日志可按 TraceID 过滤，但调用链仍需手动查日志，缺乏可视化工具。

### 优化方案

接入 OpenTelemetry + Jaeger，将 TraceID 贯穿完整调用链并可视化展示。

**Span 划分：**

```
gRPC Execute [traceID=abc123]
    └── P2C 负载均衡选节点 [span]
    └── WebSocket Send [span]
    └── Worker 容器执行 [span]
        └── 文件创建 [span]
        └── 容器内执行 [span]
    └── HTTP 回调 callBackUrl [span]
```

**与 Prometheus 联动（Exemplars）：**

Histogram 指标中附加 TraceID，从 Grafana 的异常请求直接跳转到 Jaeger 的完整调用链，实现指标→链路的联动排查。

### 预期收益

- P99 延迟可定位到具体执行环节
- 异常请求一键溯源，排查效率大幅提升

## 实施路线建议

```
第一阶段（1-2 周）
├── 结果缓存           ← 低成本，高收益
├── 接口限流防护       ← 安全基线，成本低
└── 代码预检查（基础版）← 安全兜底

第二阶段（2-4 周）
├── 容器池             ← 性能核心
└── 热更新配置         ← 运维基础

第三阶段（1-2 月）
└── 异步执行队列       ← 架构升级

第四阶段（持续迭代）
├── WebSocket 连接池
└── 分布式追踪完善
```

## 与 go-zero 迁移的关系

部分优化方案可在 go-zero 迁移过程中一并实现，避免重复建设：

| 优化项 | go-zero 内置能力 |
|--------|----------------|
| 热更新配置 | `conf` 包支持动态加载 |
| 分布式追踪 | OpenTelemetry 自动注入 |
| 异步队列 | 需结合 Redis/MQ，go-zero 不内置 |
| 容器池 / 结果缓存 | 业务层实现，与框架无关 |

建议优先实施**容器池**和**结果缓存**（与框架无关），再推进 go-zero 迁移；追踪随迁移一并完成。
